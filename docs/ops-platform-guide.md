# 运维平台使用与评分指南

运维平台负责回答一个朴素但关键的问题：这个系统现在到底靠不靠谱？

入口：

```text
http://localhost:3000/ops-platform
```

它不是为了多一个控制台，而是把工作空间、生成链路、基础环境、策略数据和日志这些分散事实集中起来。开发时看它可以少猜很多事；演示或上线前看它，可以知道问题是在环境、数据、Agent、页面还是验证链路。

## 运维平台看什么

| 页面分区 | 主要问题 |
| --- | --- |
| 工作空间健康 | 生成项目是否缺产物、验证是否通过、是否还有队列卡住 |
| 生成链路观测 | 当前请求走到哪一步、最后失败在哪个阶段、修复是否触发 |
| 基础环境 | Node、npm、Agent CLI、数据库、TimescaleDB、market-data、Redis、Loki 是否可用 |
| 日志 | Loki 集中日志、本地 Next 日志、评测队列日志和运行日志 |
| 策略健康 | 股票池覆盖、策略目录、数据新鲜度、扫描准备度 |

如果你只想知道“页面为什么打不开”，先看基础环境。如果你想知道“某个生成工作空间为什么不可用”，先看工作空间健康。如果你想知道“为什么策略平台结果不可信”，先看策略健康和数据覆盖。

## 评分不是玄学

运维平台的评分是用于定位风险的工程评分，不是业务价值评分。它的原则是：阻断项扣得重，warning 扣得轻，缺少可选组件在 `auto` 模式下不直接让系统不可用。

### 工作空间健康分

单个工作空间从 100 分开始：

| 项 | 扣分 |
| --- | --- |
| 阻断项 | 每个 `18` 分 |
| 风险项 | 每个 `4` 分 |

阻断项通常来自缺少关键产物、自动验证失败、产物契约失败或视觉检查失败。风险项通常来自数据质量 warning、验证 warning、视觉 warning、未知状态产物或仍在运行的队列。

工作空间健康状态按最严重结果收敛：

| 状态 | 含义 |
| --- | --- |
| `healthy` | 关键产物齐全，验证和视觉检查没有阻断 |
| `warning` | 可以打开，但有数据质量、视觉或队列风险 |
| `failed` | 存在明确阻断项，需要修复后再交付 |
| `unknown` | 产物或验证报告不足，还不能判断 |

### 运维总评分

运维总览有两层。

第一层是检查项摘要：失败项每个扣 `18` 分，warning 每个扣 `7` 分，unknown 每个扣 `3` 分。

第二层是健康画像，按权重聚合：

| 画像 | 因子 | 权重 |
| --- | --- | --- |
| 运行健康 | 核心服务 | 40 |
| 运行健康 | 工具链 | 20 |
| 运行健康 | 存储与表结构 | 25 |
| 运行健康 | 可观测性 | 15 |
| 项目健康 | 平均交付分 | 45 |
| 项目健康 | 健康占比 | 30 |
| 项目健康 | 阻断风险 | 25 |
| 策略健康 | 策略目录 | 25 |
| 策略健康 | 股票池覆盖 | 30 |
| 策略健康 | 数据新鲜度 | 20 |
| 策略健康 | 回测准备度 | 15 |
| 策略健康 | 研究数据源 | 10 |

分数区间：

| 分数 | 状态 |
| --- | --- |
| `>= 85` | 正常 |
| `70 - 84` | 有风险 |
| `< 70` | 需要处理 |

## 日志怎么读

日志页会优先查 Loki。Loki 不可用时，会按降级配置读取本地日志文件，例如 `.next/dev/logs/next-development.log`、`tmp/runtime/*.log` 和评测队列日志。

建议按这个顺序筛选：

1. 先筛时间范围，缩到问题发生前后 10 到 30 分钟。
2. 再按来源筛选，例如 Server、Browser、market-data、eval 或 docker。
3. 再搜关键词，例如 `error`、`failed`、`validation`、`ingestion`、`timeout`、`Reached maximum`。
4. 如果是一条 JSON 日志，先看 `source`、`level`、`message`，再看业务字段。

前端开发日志里会有很多编译成功信息，这些通常不是问题。真正值得关注的是错误栈、API 失败、SSE 断连、Agent runtime 返回的错误和验证失败摘要。

## 降级模式

运维平台要支持本地开发不完整组件的情况。

| 模式 | 适用场景 | 行为 |
| --- | --- | --- |
| `auto` | 日常开发 | 可选组件缺失给 warning，页面继续可用 |
| `strict` | CI、演示、生产巡检 | 必需组件缺失直接失败 |
| `offline` | 只看页面、只查本地文件 | 跳过市场数据后端和可观测组件探测 |

关键配置在 `.env`：

```env
QUANTPILOT_DEGRADATION_MODE=auto
QUANTPILOT_MARKET_API_REQUIRED=0
QUANTPILOT_OBSERVABILITY_REQUIRED=0
QUANTPILOT_REDIS_REQUIRED=0
```

如果用户反馈“页面打不开”，不要先把所有组件都重启一遍。先用运维平台或 `npm run doctor` 判断是哪一层缺失，再动对应组件。

## 常见排查路径

| 现象 | 先看 | 再看 |
| --- | --- | --- |
| 首页打不开 | 基础环境、Next dev 日志 | `lsof -i :3000`、`npm run dev` 输出 |
| 生成项目预览打不开 | 工作空间健康、预览端口 | 项目 `npm run build` 和预览进程 |
| 聊天页报 Agent 错误 | 生成链路观测、日志搜索 requestId | CLI 配置、模型返回、最大轮数 |
| 策略平台加载慢 | 策略健康、market-data 日志 | 股票池分页、Redis、SQL 查询 |
| 补数任务不知道是否在跑 | 策略健康、补数 job 心跳 | `quant.market_data_ingestion_jobs` |
| Loki 没日志 | 可观测性检查、Alloy 状态 | 本地日志兜底是否可读 |
| 工作空间健康分低 | 产物列表和 next actions | validation、artifact contracts、visual validation |

## 运行命令

```bash
npm run doctor
npm run doctor:full
```

启动可观测性组件：

```bash
npm run obs:up
npm run obs:logs
```

只想在没有外部组件时做本地排查：

```bash
QUANTPILOT_DEGRADATION_MODE=offline npm run doctor
```

## 运维平台未来还可以增强什么

短期最有价值的是：

- 把补数、评测和生成队列的进度统一成可暂停、可恢复、可追踪的 job 模型。
- 为关键 API 增加耗时和错误率指标，让 Grafana 不只看日志。
- 把工作空间健康快照写入 PostgreSQL，支持趋势对比。
- 将大日志、截图和长报告接对象存储，数据库只保存索引。
- 在日志页支持保存筛选条件，方便复盘同一类问题。
